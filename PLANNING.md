# 1. Tech Stack & Architecture

## 1.1 Recommended Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Frontend | **Next.js (React, TypeScript) + Tailwind + shadcn/ui** | Fast UI, mobile-first, dark/light |
| Backend API | **Node.js + NestJS (TypeScript)** | Structured modules, DI, RBAC guards |
| Realtime | **Socket.IO** | Project chat, notifications, presence |
| Database | **PostgreSQL** | Best fit for multi-tenant relational data |
| ORM | **Prisma** | Type-safe, fast iteration |
| Auth | **JWT (access + refresh) + argon2/bcrypt** | Matches PRD security list |
| File storage | **AWS S3 / Cloudflare R2** (hackathon: local disk) | Secure file access via signed URLs |
| Validation/KYC | Metadata in DB; files in object storage | Admin reviews in dashboard |
| Email | **Resend / SMTP** | Invitations, OTP, verification |
| Hosting | **Vercel (FE) + Render/Fly (API) + Neon (Postgres)** | Zero-config, free tiers |

> **Hackathon shortcut:** Use Next.js route handlers for the backend instead of a
> separate NestJS service. Split into NestJS in the Startup Track.

# 2. Data Model (Core Entities)

Tenant isolation is enforced by a `workspaceId` foreign key on most tables (§5).

| Entity | Key Fields | Relations |
|--------|-----------|-----------|
| **User** | id, email (unique), passwordHash (argon2), phone, role, verificationStatus, kycStatus, mfaSecret | belongs to Workspaces via Membership |
| **Workspace** | id, ownerId, type, name, slug (unique per tenant), plan | has Memberships, Projects, Documents |
| **Membership** | id, workspaceId, userId, roleInWorkspace, invitedBy, status | User ↔ Workspace |
| **Invitation** | id, workspaceId, token (random 32+ bytes), email, role, expiresAt, usedAt, singleUse | generated by owner |
| **KycSubmission** | id, userId, type, docs[] (encrypted keys), status, reviewedBy, reviewedAt | reviewed by Super Admin |
| **Project** | id, workspaceId, title, status, priority, timeline, clientId, milestones[] | belongs to Workspace |
| **Milestone** | id, projectId, title, dueDate, status | belongs to Project |
| **Document** | id, workspaceId, projectId?, uploadedBy, fileKey, mime, version, versionHistory[] | belongs to Workspace |
| **Message** | id, workspaceId, projectId?, senderId, body (sanitized), createdAt | realtime + persisted |
| **ActivityLog** | id, workspaceId, actorId, action, entityType, entityId, createdAt | audit trail |
| **Notification** | id, userId, type, payload, readAt | in-app + email |
| **Session** | id, userId, refreshTokenHash, device, ip, expiresAt, revokedAt | revocable sessions |

# 3. Feature Breakdown (MVP)

## 3.1 Authentication
- Email + password (argon2id hashing, constant-time compare)
- JWT access (15 min) + refresh token (7 days, rotated, stored hashed)
- Email verification link (signed, expiring token)
- OTP verification (email; SMS later) — rate-limited per account
- Password reset (signed token, single-use, short TTL)
- **Optional MFA (TOTP)** for Super Admin and Company owners

## 3.2 KYC & Verification
- Company: reg number, license, certificate, owner ID, address proof
- Freelancer: gov ID, skills, portfolio
- Client: verification on invite acceptance
- Super Admin dashboard: approve / reject (audit-logged)
- Status gates workspace access

## 3.3 Multi-Tenant Workspaces
- Each Company/Freelancer gets one workspace
- Invitation links scoped to inviter's workspace only
- All queries filtered by `workspaceId` (DB constraint + middleware)
- Isolation covered by automated tests

## 3.4 Projects & Documents
- Project: create, status, priority, timeline, team, client, milestones, deadlines
- Documents: upload (PDF/images/ZIP/Word/Excel/video), preview, download via signed URLs, version history

## 3.5 Communication (Socket.IO)
- Real-time project chat (room = workspace/project)
- Project comments, announcements, presence
- In-app + email notifications over socket

```
client → server: join_workspace, send_message, typing, mark_read
server → client: message:new, notification:new, presence:update, typing
```

## 3.6 Activity Timeline & Profiles
- Every mutating action writes an `ActivityLog` row
- Profile: photo, contact, verification badge, company details, social links

# 5. Security — Attack-Resistant Design

> Goal: make the platform resilient to the **OWASP Top 10** and common SaaS attacks.
> Defense in depth: every layer (network, app, data, tenant) has controls.

## 5.1 Threat Model

| Threat | Vector | Impact | Primary Defense |
|--------|--------|--------|-----------------|
| Credential theft / brute force | Login endpoint | Account takeover | Argon2id, rate limit, lockout, MFA |
| Token theft / replay | JWT interception | Impersonation | Short-lived access, HTTPS-only, `HttpOnly` refresh cookie |
| Tenant escalation / data leak | Forged `workspaceId` | Cross-tenant breach | `TenantContext` middleware, DB FK + tests |
| IDOR (insecure direct object ref) | Guess `/project/123` | Data exposure | Ownership checks on every resource |
| XSS | Unsanitized chat/doc | Session hijack | Output encoding, CSP, sanitize input |
| SQL Injection | Untrusted query params | Full DB compromise | Parameterized queries (Prisma ORM) |
| CSRF | Cross-site form post | State change | SameSite cookies, CSRF tokens on mutations |
| File upload abuse | Malicious file | RCE / storage abuse | Type/size allowlist, virus scan, signed URLs |
| Invitation abuse | Leaked/token guess | Unauthorized join | Random 32+ byte token, single-use, TTL |
| DDoS / abuse | Traffic flood | Downtime | Rate limiting + WAF + autoscaling |
| Privilege escalation | Role manipulation | Admin takeover | Server-side RBAC guards, no client trust |
| Sensitive data exposure | Logs/errors | Leak | No secrets in logs, generic error messages |

## 5.2 Defensive Controls (Implementation)

### Transport & Network
- **TLS 1.2+ everywhere** (WSS for sockets), HSTS, secure headers
- **WAF** (Cloudflare/managed) + DDoS protection
- **CSP, X-Frame-Options (DENY), X-Content-Type-Options (nosniff)**
- **CORS** restricted to known origins only

### Authentication & Sessions
- **Argon2id** password hashing, constant-time comparison
- **JWT access (15 min) + refresh (7d, rotated)**; refresh stored as **hash only**,
  `HttpOnly` + `Secure` + `SameSite=Strict` cookie; revocable (`Session` table)
- **Account lockout** after N failed attempts; progressive backoff
- **MFA (TOTP)** for Super Admin + Company owners
- **Logout-all** and per-device session revocation

### Authorization & Multi-Tenancy
- **RBAC guards** on every route (SUPER_ADMIN/COMPANY/FREELANCER/CLIENT)
- **`TenantContext` middleware** injects `workspaceId` from JWT and forces it on all
  queries — users can never query outside their workspace
- **Ownership/IDOR checks** on every resource (project, doc, message)
- DB-level: foreign keys to workspace + partial unique indexes

### Input & Output
- **Strict server-side validation** (zod/class-validator) on all inputs
- **Parameterized queries** via Prisma (no string-built SQL)
- **Output encoding + HTML sanitization** (e.g., DOMPurify) for chat/comments
- **Generic error messages**; stack traces never sent to client; secrets never logged

### File Security
- **Allowlist** of MIME types + max size; reject double extensions
- **Virus/malware scan** before storing (Startup Track)
- Stored with **random keys**, **encrypted at rest (SSE-KMS)**
- Served only via **short-lived signed URLs**, scoped to workspace + user permission

### Invitations
- Token = **`crypto.randomBytes(32)`**, **single-use**, **TTL (e.g., 7 days)**
- Bound to specific email; reused/guessed tokens invalid
- Revocable by inviter

### Rate Limiting & Monitoring
- **Per-IP + per-user** throttling (e.g., 100 req/min; stricter on auth/OTP)
- **Audit logging** of all mutations (`ActivityLog`)
- **Anomaly alerting** (multiple failed logins, bulk downloads)
- **Dependency scanning** (npm audit) + secret scanning in CI

### Secrets & Config
- All secrets in env/secret manager (never in code or logs)
- Separate keys per environment; rotation policy

# 6. Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| KYC fraud / fake companies | High | Manual review + 3rd-party verification vendor |
| Workspace data leak | High | TenantContext + isolation tests + FK constraints |
| Realtime scaling | Med | Socket.IO Redis adapter + sticky sessions |
| Scope creep in hackathon | High | Strict demo scope; stub the rest |
| Demo stub shipped to prod | High | Clearly flag stubs; never promote without hardening |
| Token/session theft | High | Short TTL, HttpOnly cookies, MFA, revocation |
